## 1. 产品总述

### 1.1 产品名称

建议产品名称：

```text
OpenHarmony 社区电动车安全充电终端
```

比赛展示名称：

```text
基于 OpenHarmony 的社区电动车安全充电南向控制终端与北向监管系统
```

产品化名称可选：

```text
安充云控终端
社区智能安全充电盒
OpenHarmony 四口安全充电终端
```

### 1.2 产品定位

本产品面向社区、校园、园区、物业集中停放场景，设计一套基于 RK2206 OpenHarmony 开发板的四口低压安全充电终端原型。

产品以南向硬件开发为核心，通过继电器、功率检测、插枪检测、车端电压检测、身份认证、声光语音告警等硬件模块，实现从预约、认证、插枪、预检、充电、充满、占位、断电、复位到事件追溯的完整闭环。

北向平台负责监管、预约、告警记录、数据可视化、数字孪生和 AI 运维问答。北向平台不是作品主角，而是承接端侧数据，让南向硬件能力被完整展示出来。

### 1.3 产品一句话

```text
用 OpenHarmony 做一个能真实感知、真实控制、真实断电、可追溯、可监管的社区电动车安全充电终端。
```

### 1.4 作品核心边界

本项目是比赛原型，不直接接入 220V 市电，不直接给真实电动车电池充电。

演示方式：

```text
5V/12V 低压负载
+ 4 路继电器
+ INA226 功率检测
+ 插枪检测
+ 电池电压模拟
+ 北向平台
```

这样既能安全展示真实嵌入式控制逻辑，又能避免高压电气风险。

## 2. 产品目标

### 2.1 比赛目标

本作品参加嵌入式开发大赛，评分重点应放在以下内容：

1. OpenHarmony 端侧开发能力。
2. GPIO、I2C、UART、WiFi/MQTT 等南向接口综合使用能力。
3. 多外设驱动和抽象能力。
4. 继电器真实通断控制能力。
5. 电压、电流、功率真实检测能力。
6. 状态机、故障码、事件日志设计能力。
7. 北向监管平台对端侧数据的完整承接能力。
8. 作品实物完整度和演示稳定性。

### 2.2 产品目标

产品层面需要实现：

```text
一套四口社区充电终端原型
一个完整硬件精测口
三个业务场景展示口
```

### 2.3 技术目标

南向硬件目标：

- RK2206 控制 4 路继电器。
- RK2206 读取 INA226 电压、电流、功率。
- RK2206 检测插枪状态。
- RK2206 读取车端电压模拟值。
- RK2206 控制 RGB、蜂鸣器、语音模块。
- RK2206 通过 WiFi/MQTT 上报数据。
- RK2206 接收北向预约和控制命令。

北向平台目标：

- 展示 4 个充电口实时状态。
- 支持预约充电。
- 支持告警和故障码查询。
- 支持充电记录查看。
- 支持数字孪生动画。
- 支持 AI 解释断电原因。

## 3. 行业背景与痛点

### 3.1 社区电动车充电痛点

| 痛点 | 具体表现 | 产品解决方式 |
|---|---|---|
| 私拉乱接 | 居民从楼上飞线充电，存在火灾风险 | 规划集中充电终端，低压原型模拟安全充电 |
| 充电器质量不一 | 劣质充电器可能过热、过流 | 端侧检测电压、电流、功率、温度 |
| 充满不拔 | 长期占用充电口，影响其他用户 | 充满判断、语音提醒、占位告警 |
| 监管滞后 | 物业不知道哪个端口异常 | MQTT 上报、后台告警、事件记录 |
| 事故难追溯 | 只有结果，没有数据证据 | 故障码、事件快照、安全证据链 |
| 预约难 | 高峰时段抢位、排队混乱 | 北向预约，板端执行预约规则 |
| 设备状态不可见 | 用户不知道能不能充、是否满电 | LCD、RGB、后台、数字孪生同步显示 |

### 3.2 行业趋势

根据 2026 年发布的《电动自行车充电桩安全技术规范》国家标准信息，该标准号为 GB 47371-2026，发布日期为 2026-03-31，实施日期为 2027-04-01，标准类别为安全。该方向说明电动自行车充电桩正在进入更规范的安全技术要求阶段。

新华社在 2026-01-07 对该强制性国标报批稿的报道中提到，电动自行车充电桩应具有短路、过流、输入过压、空载、充满断开、最长充电时间、超温等保护措施。

因此，本作品应围绕这些行业安全能力展开，而不是只做一个简单断电演示。

### 4.3 本项目差异化

与成熟商用品相比，本项目不是追求商用规模，而是突出嵌入式开发大赛中的可解释、可演示、可验证。

本项目特色：

1. 基于 OpenHarmony RK2206 板端真实运行。
2. 不是纯网页或纯仿真，有继电器真实低压断电。
3. 不是单点断电，覆盖预约、认证、插枪、预检、充电、充满、占位、断电、复位。
4. 不是只显示数据，而是把数据参与状态机和风险评分。
5. 不是只说 AI，而是让 AI 根据故障码和事件快照解释原因。
6. 不是只做一个口，而是用 4 口模型体现社区场景。

## 7. 硬件产品方案

### 7.1 开发板使用方案

#### 7.1.1 主控开发板

主控板：

```text
RK2206 OpenHarmony 综合开发板
```

用途：

- 主状态机。
- LCD 显示。
- 按键输入。
- WiFi/MQTT 通信。
- GPIO 控制继电器、RGB、蜂鸣器。
- I2C 读取 INA226。
- UART 连接语音模块或 RFID。

选择理由：

- 已经跑通 OpenHarmony 编译、烧录和基础板端流程。
- 适合体现 OpenHarmony 端侧嵌入式开发能力。
- 支持 WiFi/MQTT，便于打通北向。

#### 7.1.2 可选扩展开发板

| 开发板 | 是否当前使用 | 用途 |
|---|---|---|
| LZ3863 / 星闪小板 | 后期可选 | 多节点或无线子端口扩展 |
| 第二块 OpenHarmony 板 | 后期可选 | 子节点采集或分布式演示 |
| STM32 小板 | 暂不建议 | 会分散 OpenHarmony 主线 |

当前阶段不建议同时接入多块开发板。主线应先把 RK2206 做扎实。

### 7.4 低压电气设计

低压演示推荐：

```text
主控侧：RK2206 由开发板供电
负载侧：独立 5V 或 12V 电源
继电器侧：5V 继电器模块
负载：小灯、小风扇、小电机、电阻负载
保护：急停 + 单口保险丝 + 端子排
```

关键原则：

- 继电器控制低压正极线路。
- 不使用真实 220V 插座。
- 充电口外观使用低压 DC 插头或自制充电枪。
- RK2206 与外部电源共地前必须确认电压。
- 高电流负载不要直接由 RK2206 供电。

### 7.5 实物结构设计

建议结构：


## 8. OpenHarmony 端侧软件方案

### 8.1 软件分层

```text

-> 设备抽象层
-> 数据模型层
-> 安全策略层
-> 业务状态机层
-> 通信与展示层
```

### 8.2 硬件驱动层

```text

```

每个驱动只做一件事：

- 初始化硬件。
- 读取输入。
- 控制输出。
- 返回错误码。

### 8.3 设备抽象层

统一接口：

```c
int power_sensor_read(uint8_t port_id, PowerData *data);

```

目的：

- 后续替换硬件时不影响业务状态机。
- 体现 OpenHarmony/HDF 风格的南向设备抽象思想。

### 8.5 状态机

状态：

```text
IDLE
RESERVED
AUTH
PLUG_CHECK
PRECHECK
CHARGING
FULL
OCCUPIED
ALERT
CUTOFF
LOCKED
RESET
```

状态流转规则：

```text
IDLE -> RESERVED：收到预约任务
IDLE -> AUTH：用户直接认证
RESERVED -> AUTH：到达预约时间
AUTH -> PLUG_CHECK：认证通过
PLUG_CHECK -> PRECHECK：插枪到位
PRECHECK -> CHARGING：桩端和车端电压正常
CHARGING -> FULL：满足充满条件
FULL -> OCCUPIED：超时未拔枪
CHARGING -> ALERT：轻中度风险
ALERT -> CUTOFF：风险升级
CHARGING -> CUTOFF：拔枪、过流、急停、烟雾
CUTOFF -> RESET：人工或管理员复位
LOCKED -> IDLE：管理员解锁
```

### 8.6 安全策略

#### 8.6.1 预检规则

进入 CHARGING 前必须通过：

```text
用户已认证
预约有效或无需预约
插枪到位
桩端电压在安全范围内
车端电压在允许范围内
继电器当前为 OFF
无未清除故障
```

#### 8.6.2 充满规则

建议组合条件：

```text
vehicle_voltage >= FULL_VOLTAGE_THRESHOLD
&& charge_current <= FULL_CURRENT_THRESHOLD
&& 持续时间 >= FULL_HOLD_SECONDS
```

这样比单独用电压或电流更可靠。

#### 8.6.3 断电规则

立即断电条件：

- 急停按下。
- 充电中拔枪。
- 烟雾异常。
- 电流超过危险阈值。
- 桩端电压严重异常。
- 车端电压严重异常。
- 继电器反馈异常。

断电后：

```text
relay_off
motor_off
rgb_red_on
buzzer_long_beep
voice_cutoff
event_upload
require_manual_reset
```

### 8.7 故障码

| 故障码 | 含义 | 触发来源 | 动作 |
|---|---|---|---|
| F001 | 未插枪启动 | plug_driver | 禁止启动 |
| F002 | 用户未授权 | auth_module | 禁止启动 |
| F003 | 预约时间未到 | reservation_module | 禁止启动 |
| F004 | 预约用户不匹配 | reservation_module | 禁止启动 |
| F101 | 桩端过压 | ina226/adc | 禁止启动或断电 |
| F102 | 桩端欠压 | ina226/adc | 禁止启动 |
| F201 | 车端电压过低 | adc | 禁止启动 |
| F202 | 车端电压过高 | adc | 禁止启动或断电 |
| F301 | 充电过流 | ina226 | 告警并断电 |
| F302 | 功率异常波动 | ina226 | 告警 |
| F401 | 充电中拔枪 | plug_driver | 立即断电 |
| F501 | 温度异常 | sensor | 告警或断电 |
| F502 | 烟雾异常 | sensor | 立即断电 |
| F601 | 继电器状态异常 | relay_driver | 故障锁定 |
| F701 | 充满占位超时 | state_machine | 上报占位 |
| F801 | 网络离线 | mqtt_module | 本地继续保护 |

### 14.1 单模块测试

| 模块 | 测试项 | 通过标准 |

| 继电器 | 单口通断 | 负载开关稳定 |
| INA226 | 电压/电流读取 | 数值随负载变化 |
| 插枪检测 | 插入/拔出 | 状态准确切换 |
| RGB | 各状态颜色 | 与状态表一致 |
| 蜂鸣器 | 短响/长响 | 告警可听 |
| MQTT | 状态上报 | 平台收到 JSON |
| 预约 | 命令下发 | 板端保存并显示 |

### 14.2 场景测试

| 场景 | 步骤 | 预期 |
|---|---|---|
| 正常充电 | 认证、插枪、预检、启动 | 继电器闭合，状态 CHARGING |
| 未插枪启动 | 认证后不插枪启动 | 禁止启动，故障 F001 |
| 预约未到 | 提前启动预约口 | 禁止启动，故障 F003 |
| 车端电压异常 | 调低或调高模拟电压 | 禁止启动或断电 |
| 充电过流 | 增大负载或模拟超流 | ALERT/CUTOFF |
| 充电中拔枪 | 拔出检测开关 | 立即断电，F401 |
| 充满 | 电压高且电流低持续 | FULL，继电器断开 |
| 占位 | FULL 后不拔枪 | OCCUPIED，事件上报 |
| 管理员复位 | 管理员认证复位 | LOCKED 返回 IDLE |

### 14.3 演示稳定性测试

演示前至少重复 10 次完整流程：

```text
预约 -> 认证 -> 插枪 -> 预检 -> 充电 -> 告警 -> 断电 -> AI 解释 -> 复位
```

要求：

- 不死机。
- 不断网。
- 不出现 LCD 乱码。
- 不出现继电器误动作。
- 不出现线束松动。

## 17. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| INA226 调不通 | 功率检测失败 | 保留模拟数据模式，继续调试真实模块 |
| GPIO 不够 | 多口控制受限 | 减少部分 RGB 独立控制，使用 IO 扩展 |
| 继电器触发不稳 | 断电展示失败 | 独立供电，加驱动管，降低负载 |
| 插枪结构不稳定 | 误判 | 先用微动开关，机械结构固定 |
| 北向平台拖慢进度 | 南向主线受影响 | 北向先做最小页面 |
| AI 问答不稳定 | 演示失败 | 准备规则模板回答 |
| 视频录制时硬件异常 | 线上展示受影响 | 提前录制备用完整视频 |
| 文档讲得像网页项目 | 评分偏离 | 答辩反复强调南向硬件和 OpenHarmony |

## 19. 答辩表达重点

### 19.1 开场

```text
我们做的不是一个简单的断电保护演示，而是基于 OpenHarmony 的社区电动车安全充电终端原型。
它以 RK2206 为端侧主控，围绕真实社区充电场景，实现预约、认证、插枪、预检、充电、充满、占位、断电和追溯闭环。
```

### 19.2 南向硬件表达

```text
南向硬件部分，我们接入了继电器、功率检测模块、插枪检测、车端电压检测、声光提示和低压负载。
其中继电器用于真实断电，INA226 用于真实读取电压、电流和功率，插枪检测和车端电压用于充电前安全预检。
```

### 19.3 OpenHarmony 表达

```text
OpenHarmony 在本项目中承担端侧设备控制、状态机管理、传感器数据采集、通信上报和远程命令校验。
我们按照驱动层、设备抽象层、数据模型层、安全策略层和通信展示层组织板端软件，体现了嵌入式系统工程化设计。
```

### 19.4 行业对标表达

```text
当前行业智能充电桩普遍强调充满自停、过流保护、远程监控、扫码刷卡和运营管理。
我们的作品在低压安全原型中对标这些能力，并进一步突出 OpenHarmony 南向开发、事件证据链和 AI 运维解释。
```

### 19.5 创新总结

```text
本项目的创新点是把社区充电安全问题从单点告警扩展为完整业务闭环：
充电前有预约和预检，充电中有功率监测和风险评分，充满后有断电和占位治理，异常后有事件追溯和 AI 解释。
```

## 21. 最终落地原则

最终开发时遵守三句话：

```text
先做硬件真实闭环，再做北向展示。
先做 1 号口深度，再做 4 口规模。
先保证演示稳定，再增加 AI 和 3D。
```

作品最小成功形态：

```text
1 号口能真实完成插枪、预检、继电器闭合、功率检测、告警断电、事件上报。
```

作品理想完成形态：

```text
4 个端口分别展示精测、预约、异常、占位四类社区场景，
并通过 OpenHarmony RK2206、北向平台、数字孪生和 AI 运维形成完整产品闭环。
```
